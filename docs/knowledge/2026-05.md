## Architecture Patterns

- 2026-05-07 — **Codex command discoverability needs a bridge skill, not just a command doc.** Keep the canonical workflow in `commands/<name>.md`, then add a thin `skills/<name>/SKILL.md` bridge so Codex's live plugin reader can surface it in the picker/import path. Do not duplicate the workflow body in the bridge.

## Known Gotchas

- 2026-05-07 — **Alias names need their own bridge files if you want them discoverable.** A legacy markdown alias such as `skills/codex-delegate.md` is not enough for the picker path. Use a real `skills/<alias>/SKILL.md` bridge when discoverability matters.

## Key Decisions

- 2026-05-07 — **Historical note: the earlier packaging pass treated the command doc as the source of truth.** That model has since been superseded by the skill-first wrapper pattern below.
- 2026-05-08 — **Skills are the canonical instruction bodies; commands are wrappers.** The stop/history workflows now live in `skills/<name>/SKILL.md`, and `commands/<name>.md` only point back to them for slash-command discoverability.
- 2026-05-08 — **The earlier command-first note is historical context, not the current architecture.** The current Sidekick packaging model intentionally inverts that relationship so the skill remains canonical and the command stays thin.
- 2026-05-08 — **The docs tree now has explicit homes for public docs, internal audits, workflows, and archived spec/design material.** Keep the gateway index aligned with the actual tree so the docs-scheme stays truthful as the repo grows.
- 2026-05-08 — **Public help docs must follow local asset paths, not root-relative placeholders.** Use existing repo assets for icons and keep the search index relative so the docs still work when deployed from the `docs/` tree.
