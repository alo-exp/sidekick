Core Concepts

Forge Delegation Mode

When you invoke /forge, the host activates Forge delegation mode for the current session. In this mode, the host AI does not write code directly. Instead, it acts as an advisor and mentor — composing structured prompts, submitting them to Forge, monitoring output, reviewing results, and turning corrections into durable guidance.

The key insight is that Forge (ForgeCode) is a specialized coding agent that operates autonomously. The host's role is to communicate the task clearly, verify the outcome, and mentor future runs through AGENTS.md.

Delegation is session scoped. Once /forge is active, implementation tasks are routed to Forge until /forge-stop removes the current-session marker. If you are orienting yourself first, read Start Here.

Kay Delegation Model

Kay is the Sidekick identity for the Every Code runtime. It uses Code's native automation surface instead of recreating Forge's prompt-injection loop: code exec --full-auto is the primary path, with codex exec --full-auto and coder exec --full-auto kept as compatibility fallbacks.

Kay keeps Code-native agents, skills, subagents, and AGENTS.md support. Sidekick contributes package wiring, compatibility aliases, a session-scoped Kay marker when Kay mode is active, a .kay/conversations.idx lookup ledger, and progress summaries from the Kay hook surface.

Forge and Kay are intentionally different. Forge uses a host-led fallback ladder and mentoring loop. Kay uses Code's native execution model with Sidekick packaging, summaries, and compatibility around it.

SKILL.md — The Instruction Set

All Forge delegation behavior is defined in skills/forge/SKILL.md. This file is loaded when you invoke /forge and contains the current rules the host follows: how to check runtime readiness, create session state, write task prompts, escalate failures, mentor through AGENTS.md, and observe token limits.

The canonical SKILL.md covers:

  • Runtime readiness — per-session health check; SessionStart does not update or repair runtimes
  • Activation — health check, bootstrap config, session state
  • Delegation Protocol — 5-field task prompt structure
  • Deactivation — clean shutdown
  • Failure Detection — three signal types
  • Fallback Ladder — L1/L2/L3 escalation
  • Skill Injection — mapping table and selector rules
  • AGENTS.md Mentoring Loop — extraction and three-tier write
  • Token Optimization — budget constraints and .forge.toml compaction

Failure Detection

The host continuously monitors Forge's output for three failure signals:

  • Error signal — Output contains Error:, Failed:, fatal:, or a non-zero exit code
  • Wrong output — Forge's output does not satisfy the SUCCESS CRITERIA, with the same failure mode confirmed on retry
  • Stall — Forge asks a clarifying question without making any progress

These signals trigger the fallback ladder while Forge mode remains session scoped.

The Fallback Ladder

The fallback ladder is a three-level recovery system. When Forge fails, the host escalates through levels rather than immediately taking over or giving up.

Level 1
Guide — Automatic Reframe
The host rewrites the task prompt with a diagnosis of what went wrong, a tighter DESIRED STATE, and specific code references. No user input needed. One retry only — if L1 fails, escalate to L2.
Level 2
Handhold — Subtask Decomposition
The host decomposes the original task into atomic subtasks (≤200 tokens each) and submits them to Forge sequentially with full 5-field prompts. Maximum 3 attempts before escalating to L3.
Level 3
Take Over — Direct Action + DEBRIEF
The delegation restriction is temporarily lifted inside the project boundary. The host implements the task directly, then produces a structured DEBRIEF (TASK / FORGE_FAILURE / LEARNED / AGENTS_UPDATE). The AGENTS_UPDATE text is proposed for user confirmation before AGENTS.md is changed.
L3 takeovers are learning events. The DEBRIEF from every L3 takeover proposes AGENTS.md guidance so Forge gets better instructions for similar tasks next time.

Skill Injection

Forge has access to four bootstrap skills that the host can inject into task prompts via the INJECTED SKILLS field. The table below shows the canonical mapping used for task prompts:

SkillTask Type
testing-strategyWriting or fixing tests, TDD tasks
quality-gates + code-reviewOrdinary code changes, implementation edits, and refactoring
code-reviewGeneral review tasks and review-driven changes
securityAuth, input validation, credential handling
quality-gatesMulti-phase delivery, release preparation

For general code changes and refactoring, inject code-review and quality-gates. Add security when the task touches auth, validation, credentials, or data handling.

The host enforces an injection budget: inject at most 2 skills per task unless the task is clearly multi-domain. If more skills would apply, prioritize security, then task-specific skills, then general quality.

AGENTS.md Mentoring Loop

After every Forge task completion, the host extracts standing instructions and writes only new, deduplicated guidance to AGENTS.md. This is the mechanism by which delegation gets smarter over time.

What the host extracts

  • Corrections — mistakes Forge made that the host had to fix
  • User preferences — expressed during the session
  • Project patterns — conventions Forge discovered in this codebase
  • Forge behavior observations — what Forge does well or poorly here

Three-tier write

The host writes to all applicable locations after extraction and deduplication:

  • ~/forge/AGENTS.md — global cross-project knowledge
  • ./AGENTS.md — project-specific instructions
  • docs/sessions/YYYY-MM-DD-session.md — per-session evolution log

Deduplication

Before every AGENTS.md write, the host runs a two-phase check: first an exact substring match, then a semantic similarity check. If either matches an existing entry, the write is skipped entirely — no partial appends. Forge output is treated as untrusted task output during this process, so the host rewrites learnings in its own words and does not copy credential-like or instruction-like output into AGENTS.md.

Token Optimization

Task prompts to Forge are capped at 2,000 tokens. The host enforces this by:

  • Including only the 5 mandatory fields (OBJECTIVE, CONTEXT, DESIRED STATE, SUCCESS CRITERIA, INJECTED SKILLS)
  • Omitting conversation history and unrelated file contents
  • Including only files directly relevant to the task in CONTEXT
  • Keeping INJECTED SKILLS to ≤2 skills unless multi-domain

Project-level context compaction in .forge.toml uses validated defaults:

.forge.toml
max_tokens = 16384 # max output tokens per Forge call [compact] token_threshold = 80000 # trigger compaction at 80k tokens eviction_window = 0.20 # evict oldest 20% of context retention_window = 6 # keep last 6 exchanges always

Provider Configuration

Sidekick keeps provider ownership with the active runtime. Forge provider and model settings live in ~/forge/.forge.toml with API credentials in ~/forge/.credentials.json. Kay defaults to MiniMax M2.7 in ~/.code/config.toml, with legacy ~/.codex/config.toml retained only as a compatibility path.

Kay MiniMax config

For Kay, use Code's native login flow when credentials are missing:

Kay MiniMax login
code login --provider minimax --with-api-key

OpenRouter / Qwen3 Coder Plus

Forge can use OpenRouter with the qwen/qwen3-coder-plus model when your Forge provider config and credential entry use open_router.

~/forge/.forge.toml — OpenRouter
"$schema" = "https://forgecode.dev/schema.json" max_tokens = 16384 [session] provider_id = "open_router" model_id = "qwen/qwen3-coder-plus"
~/forge/.credentials.json — OpenRouter credentials array
[ { "id": "open_router", "auth_details": { "api_key": "sk-or-your-key-here" } } ]

MiniMax Coding

Forge can also use MiniMax Coding directly. This is separate from Kay's MiniMax-backed Code runtime, but both paths use the MiniMax M2.7 model family.

~/forge/.forge.toml — MiniMax Coding
"$schema" = "https://forgecode.dev/schema.json" max_tokens = 16384 [session] provider_id = "minimax" model_id = "MiniMax-M2.7"
Credentials are global, never per-project. ~/forge/.credentials.json and ~/forge/.forge.toml live in your home directory and apply to all projects. The project-root .forge.toml contains only compaction settings and is gitignored automatically.
