Delegation Skills

CommandDescription
/forgeActivate Forge delegation mode for the current session. Runs the four-part health check, bootstraps missing Sidekick-owned project files, loads SKILL.md, and arms hook enforcement through the session marker.
/forge-stopDeactivate Forge delegation mode and return to direct host implementation.
kay-delegateUse the Kay delegate workflow for kay runtime tasks. Some hosts display the plugin-prefixed name sidekick:kay-delegate. Older binary names remain compatibility-only.
/kay-stopDeactivate Kay delegation mode and return to direct host implementation.

Plugin Hooks

Sidekick ships a small SessionStart surface for first-run bootstrap and legacy cleanup, plus PreToolUse and PostToolUse hooks that stay dormant until a sidekick session marker exists. SessionStart no longer updates or repairs Forge/Kay runtimes after install.

HookEventDescription
install.shSessionStartRuns the guarded first-run plugin bootstrap when the package-local .installed sentinel is absent.
scrub-legacy-user-hooks.pySessionStartRemoves stale Sidekick-owned hook blocks from legacy user config while preserving unrelated hooks.
forge-delegation-enforcer.shPreToolUse (Write / Edit / NotebookEdit / Bash)When Forge mode is active, denies direct implementation edits, rewrites forge -p calls to inject --conversation-id, --verbose, and output prefixing, and appends an entry to .forge/conversations.idx.
codex-delegation-enforcer.shPreToolUse (Write / Edit / NotebookEdit / Bash)When Kay mode is active, denies direct implementation edits, rewrites Kay runtime calls to include --full-auto and output prefixing, and appends an entry to .kay/conversations.idx.
forge-progress-surface.shPostToolUse (Bash)Parses Forge subprocess output, strips terminal controls, redacts obvious secrets, and surfaces a compact untrusted [FORGE-SUMMARY] STATUS block when one is present.
codex-progress-surface.shPostToolUse (Bash)Parses Kay subprocess output, strips terminal controls, redacts obvious secrets, and emits a compact untrusted [KAY-SUMMARY] progress surface when a STATUS block is present.
validate-release-gate.shPreToolUse (Bash)Release-safety guard. Blocks gh release create commands unless all four quality-gate-stage-N markers are present in ~/.claude/.sidekick/quality-gate-state and the full live Forge/Kay release pyramid has passed twice.

Kay Runtime Commands

Kay uses its native CLI automation surface. When Kay mode is active, the hook treats kay as primary and keeps older binary names compatibility-only:

CommandPurpose
kay exec --full-auto "<task>"Primary non-interactive Kay execution path.
kay login --provider minimax --with-api-keyConfigure MiniMax credentials for the Kay default provider path.
legacy aliasesThe old code, codex, and coder binary names are retained for compatibility, not for new user-facing setup.

Task Prompt Fields

Every task delegated to Forge uses exactly these 5 fields, in this order:

FieldDescriptionLimit
OBJECTIVEOne-sentence statement of what Forge must accomplish. No ambiguity.1 sentence
CONTEXTOnly files directly relevant to the task. No conversation history, no unrelated files.~600 tokens
DESIRED STATESpecific, verifiable outcome. What the code/system looks like when done.~300 tokens
SUCCESS CRITERIAChecklist the host uses to verify Forge succeeded. Each item must be checkable.3–6 items
INJECTED SKILLS≤2 bootstrap skills relevant to the task type.1–2 skills

Total prompt budget: 2,000 tokens max.

.forge.toml Config

Created in the project root on first /forge invocation if absent. Contains compaction settings only — Forge API credentials live in ~/forge/.credentials.json, while provider and model config live in ~/forge/.forge.toml.

.forge.toml (project root — compaction only)
# API credentials: ~/forge/.credentials.json # Context compaction — validated defaults max_tokens = 16384 # max output tokens per call [compact] token_threshold = 80000 # trigger at 80k tokens eviction_window = 0.20 # evict oldest 20% retention_window = 6 # always keep last 6 exchanges

Global provider config (~/forge/.forge.toml)

The global config sets the provider and model. The current Forge setup path supports OpenRouter / Qwen3 Coder Plus and MiniMax Coding / MiniMax M2.7:

~/forge/.forge.toml — OpenRouter / Qwen3 Coder Plus
"$schema" = "https://forgecode.dev/schema.json" max_tokens = 16384 [session] provider_id = "open_router" model_id = "qwen/qwen3-coder-plus"
~/forge/.forge.toml — MiniMax Coding
"$schema" = "https://forgecode.dev/schema.json" max_tokens = 16384 [session] provider_id = "minimax" model_id = "MiniMax-M2.7"
.forge.toml is gitignored by default. API credentials are stored globally in ~/forge/.credentials.json, and provider/model config is stored in ~/forge/.forge.toml, not in the project root. When the host guides you through setup, it writes these files through file-write tooling rather than shelling secrets through Bash.

Forge Output Format

Every Forge task response should include a structured output block with four mandatory fields. The host reads these fields to verify success, detect failures, review assumptions, and extract AGENTS.md guidance when appropriate.

FieldDescription
STATUSSUCCESS or FAILURE. The host checks this first — a FAILURE triggers fallback handling.
FILES_CHANGEDList of every file Forge created, modified, or deleted during the task. Used for output review and session logging.
ASSUMPTIONSAny assumptions Forge made that were not explicit in the task prompt. The host surfaces these for user review.
PATTERNS_DISCOVEREDCodebase conventions, architecture patterns, or project quirks Forge noticed during the task. The host extracts these in its own words for AGENTS.md when they are new and safe to persist.
Example Forge output block
STATUS: SUCCESS FILES_CHANGED: - src/auth/middleware.ts (modified) - src/auth/middleware.test.ts (created) - src/types/auth.ts (modified) ASSUMPTIONS: - JWT secret read from process.env.JWT_SECRET - Token expiry defaults to 24h if not configured PATTERNS_DISCOVERED: - Project uses vitest, not jest - All middleware follows Express 5 async error pattern - TypeScript strict mode is enabled globally
PATTERNS_DISCOVERED feeds AGENTS.md carefully. After each task, the host reads this field, treats it as untrusted output, and writes only safe, deduplicated, action-oriented learnings to ./AGENTS.md and ~/forge/AGENTS.md.

AGENTS.md Format

AGENTS.md is written in tiers when new guidance exists: global instructions live in ~/forge/AGENTS.md, project instructions live in ./AGENTS.md, and session history is appended to docs/sessions/YYYY-MM-DD-session.md.

Global AGENTS.md
# Forge Global Instructions ## Code Style - Cross-project style rules ## Testing - Cross-project test and verification habits ## Git Workflow - Cross-project commit and release habits ## Forge Behavior - Cross-project Forge corrections
Project AGENTS.md
# Project Instructions ## Project Conventions - Repo-specific structure and naming rules ## Task Patterns - Repeated task shapes and workflow notes ## Forge Corrections - Repo-specific Forge mistakes and fixes
Session log
# docs/sessions/YYYY-MM-DD-session.md - Task name - Extracted instructions - Deduplication decisions - Tiers written
