Delegation Workflow

The complete flow for delegating a coding task through Forge or Kay — from current-session readiness through output review, task history, and the Forge AGENTS.md mentoring loop. If you need the task-first docs map, read Start Here first.

Step 1 — Activate Forge

1

Run /forge

Type /forge in the host. The host loads skills/forge/SKILL.md and executes the activation sequence:

2

Health check

The host verifies four current conditions: (1) the Forge binary exists or is on PATH, (2) forge info exits successfully and reports a provider, (3) ~/forge/.credentials.json is a non-empty array of credential entries with id and auth_details, and (4) ~/forge/.forge.toml has non-empty provider_id and model_id. Any failure halts activation with a specific repair path.

3

Bootstrap project files if absent

On first invocation only, the host creates missing Sidekick-owned project files without overwriting existing files: .forge/agents/forge.md, project .forge.toml compaction defaults, and the four .forge/skills/ bootstrap skills. If ./AGENTS.md is absent or empty, the Forge mentoring bootstrap creates the project instruction sections.

4

Confirm activation

The host writes a zero-byte Forge marker under the current host session directory and confirms: "Forge-first delegation mode activated." The hook layer stays dormant until that marker exists, then routes implementation work to Forge until you run /forge-stop.

Health check failing? See Troubleshooting → Health Check Failures for step-by-step fixes. The most common causes are a missing Forge binary, malformed ~/forge/.credentials.json, or missing provider/model values in ~/forge/.forge.toml.

Kay Workflow

Kay uses Code's native automation flow. Sidekick keeps the runtime packaged, exposes the delegate/stop skills for host pickers, and preserves a project-local Kay audit ledger. In Claude Code, the Kay workflow can launch a child Code runtime; in Code/Codex, the same skill is runtime guidance because the host already is the execution runtime.

1

Verify Code runtime

Run code --version and code exec --help. If the primary binary is unavailable, use codex or coder. If MiniMax login is missing, run code login --provider minimax --with-api-key.

2

Run the delegated task

Use code exec --full-auto "<task>" when launching a child Kay runtime. Sidekick accepts code, codex, or coder; when Kay mode is active, the hook adds --full-auto if it is missing and prefixes output with [KAY] / [KAY-LOG].

3

Review the summary

The Kay progress surface emits compact [KAY-SUMMARY] context when a STATUS block appears and preserves lookup metadata in .kay/conversations.idx. Review changed files, test output, and the final task summary before continuing.

Step 2 — Compose Task Prompt

When you describe a Forge task, the host builds a structured 5-field prompt capped at 2,000 tokens:

task prompt structure
OBJECTIVE: One-sentence statement of what Forge must accomplish CONTEXT: Only relevant files + their current state DESIRED STATE: Specific, verifiable outcome (no ambiguity) SUCCESS CRITERIA: Checklist the host uses to verify Forge succeeded INJECTED SKILLS: ≤2 bootstrap skills relevant to the task type
The host writes this for you. You describe what you need in plain language. The host translates that into the 5-field structure with the right specificity and skill selection.

Step 3 — Monitor Forge Output

The host streams Forge's output and checks for three failure signals on every response:

  • Error signal — Error:, Failed:, fatal:, or non-zero exit
  • Wrong output — SUCCESS CRITERIA not satisfied, with the same failure mode confirmed on retry
  • Stall — Forge asks questions without making progress

If no failure signal is detected and SUCCESS CRITERIA are met, review the output summary in Step 5.

Step 4 — Fallback Handling (on failure)

Failures trigger the three-level fallback ladder in sequence:

L1

Guide — Reframe and retry once

The host diagnoses the failure, rewrites the prompt with a tighter DESIRED STATE and code references, and resubmits. If this succeeds, jump to Step 5. If it fails, escalate to L2.

L2

Handhold — Decompose and sequence

The host breaks the task into atomic subtasks (≤200 tokens each) and submits them one at a time. Up to 3 subtask attempts total. If they fail, escalate to L3.

L3

Take over — host implements directly

Delegation restriction is temporarily lifted inside the project boundary. The host implements the task. After completion, it produces a DEBRIEF with TASK / FORGE_FAILURE / LEARNED / AGENTS_UPDATE fields. AGENTS_UPDATE is proposed for user confirmation before AGENTS.md is changed.

Step 5 — Output Review

Whether Forge succeeded or the host took over at L3, the host presents a structured output summary for your review. The summary is drawn from Forge's output block fields:

  • FILES_CHANGED — every file Forge created, modified, or deleted
  • ASSUMPTIONS — anything Forge inferred that wasn't explicit in the prompt (review these carefully)
  • PATTERNS_DISCOVERED — codebase conventions Forge noticed during the task
  • Test results — if Forge ran tests, pass/fail summary

Confirm the output looks correct. If something needs adjustment, describe the change in plain language — the host will compose a new 5-field delegation prompt for the follow-up task. There is no need to re-activate; delegation mode remains active.

Always review ASSUMPTIONS. If Forge made an incorrect assumption (wrong env variable name, wrong test framework, etc.), correct it now. The host can convert the correction into deduplicated AGENTS.md guidance so Forge doesn't repeat the assumption.

Step 6 — AGENTS.md Update

After each Forge task completion, the host runs the AGENTS.md mentoring loop. This is automatic for extraction and deduplication; L3 takeover updates are proposed for confirmation before being written.

What gets written

  • Successful tasks — PATTERNS_DISCOVERED and any user preferences expressed during review are extracted and written only when they are new and action-oriented
  • L3 takeovers — the DEBRIEF's AGENTS_UPDATE field captures what Forge failed on and what the correct approach is; the host asks before applying it
  • Session log — a full evolution log is appended to docs/sessions/YYYY-MM-DD-session.md, recording every task, outcome, and AGENTS.md change from the session

Deduplication

Before every write, the host runs a two-phase dedup check: exact substring match first, then semantic similarity. If either matches an existing AGENTS.md entry, the write is skipped — preventing duplicate or contradictory instructions from accumulating. Forge output is treated as untrusted data, so the host rewrites learnings in its own words.

AGENTS.md three-tier write targets
# Global — applies to all projects ~/forge/AGENTS.md # Project — applies to this repo only ./AGENTS.md # Session log — audit trail docs/sessions/YYYY-MM-DD-session.md

Step 7 — Deactivate (optional)

Forge delegation mode persists for the entire session. You can delegate as many tasks as needed without re-running /forge. To return to direct host implementation:

/forge-stop

This removes the current-session Forge marker and restores normal direct-host behavior. The .forge/conversations.idx audit ledger and any accepted AGENTS.md updates are preserved.

You don't have to deactivate between tasks. Forge delegation mode is designed to stay active for a full working session. Only deactivate if you specifically want the host to implement something directly — for example, a quick one-liner that doesn't warrant the delegation overhead.
