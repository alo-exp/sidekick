What is Sidekick?

Sidekick is a Claude Code and Codex plugin that gives the active host two implementation agents. Forge packages ForgeCode for delegated terminal work. Kay packages the Every Code agent line with MiniMax M2.7 defaults, code / codex / coder compatibility, and the OpenCode Go multi-AI path. Claude Code and Codex can both route work to Forge or Kay.

When you activate Forge delegation mode, the host stops writing code directly and instead composes structured task prompts, submits them to Forge, monitors the output, and handles failures through the fallback ladder. Kay uses Code's native execution workflow for implementation while Sidekick keeps packaging, hook enforcement, summaries, and task lookup metadata aligned.

Why use a sidekick? Forge and Kay keep implementation work in a purpose-built agent while the host stays focused on task composition, quality review, and communication.

Prerequisites

You need the following before installing Sidekick:

  • Claude Code or Codex host — required to run the Sidekick plugin surface.
  • Forge or Kay agent path — choose the agent that fits the task.
  • Provider access — Forge needs a valid Forge provider config and credential entry. Kay defaults to MiniMax M2.7 through ~/.code/config.toml, with legacy ~/.codex/config.toml compatibility.
  • Node.js 18+ — required by Claude Code.
Readiness is per session. Sidekick loads the hook surface with the plugin, then checks Forge or Kay readiness when you start delegation for the current session.

Installation

Install Sidekick through the plugin surface for the host you use:

Claude Code
/plugin install alo-labs/sidekick
Codex / Kay
codex plugin marketplace add alo-labs/sidekick

Restart the host after installation so the hook surface loads before you start a Forge or Kay session. The SessionStart hooks only run first-run bootstrap and legacy hook cleanup; Forge and Kay runtime readiness is checked when delegation starts for the current session.

Codex users should install the Codex-facing Sidekick package through the Codex plugin surface. That path exposes the shared skill bodies and Kay runtime guidance without duplicating writable state from Claude Code.

Health Check

After installing, verify the runtime for the agent path you plan to use.

Forge health check

In the host, run:

in any host session
/forge # the host activates Forge delegation mode and runs the readiness check

On activation, Forge checks four things: the Forge binary, forge info provider output, the credential array in ~/forge/.credentials.json, and non-empty provider_id / model_id values in ~/forge/.forge.toml.

A successful health check looks like:

✓ Forge binary found (~/.local/bin/forge) ✓ Provider configured (forge info exit 0) ✓ Credentials present (~/forge/.credentials.json has at least one {id, auth_details} entry) ✓ Config valid (~/forge/.forge.toml has provider_id and model_id) ✓ Forge delegation mode active
Health check failing? See Troubleshooting → Health check failures for common causes and fixes.

Kay health check

For the Kay path, verify that the Every Code runtime is reachable and has a MiniMax-backed config. In Claude Code, Kay delegation launches a child Code runtime. In Code/Codex itself, the Kay skill is packaging and runtime guidance because the host already is the execution runtime.

Code / Codex host
code --version # or codex --version / coder --version code exec --help # or codex exec --help / coder exec --help code login --provider minimax --with-api-key # if login is missing

Your First /forge Task

Once the health check passes, the host is in Forge delegation mode. Every implementation task you give the host is delegated to Forge via a structured 5-field task prompt.

  1. 1

    Describe what to build

    Tell the host what you need in plain language. It will compose the OBJECTIVE, CONTEXT, DESIRED STATE, SUCCESS CRITERIA, and INJECTED SKILLS fields automatically.

  2. 2

    Host submits to Forge

    The host sends the structured prompt to Forge (max 2,000 tokens). The PreToolUse hook injects --conversation-id, --verbose, and output prefixes automatically; do not add those flags manually.

  3. 3

    Host monitors output

    The host watches for three failure signals: error output, SUCCESS CRITERIA not met on retry, or Forge asking a clarifying question without progress.

  4. 4

    Review and confirm

    The host presents Forge's output for your review. If Forge succeeded, the task is done. If Forge failed, the fallback ladder kicked in.

First task recommendation: Start with a small, well-scoped task like "add a --verbose flag to the CLI" or "write unit tests for the auth module." This lets you observe the delegation flow without risking complex failures.

Your First Kay Task

Kay uses Code's native non-interactive execution path. Use code exec --full-auto first, then fall back to codex exec --full-auto or coder exec --full-auto when the primary binary is unavailable or conflicts with another app.

Kay delegation path
code exec --full-auto "implement the focused task, run tests, summarize changes" # fallback aliases codex exec --full-auto "implement the focused task, run tests, summarize changes" coder exec --full-auto "implement the focused task, run tests, summarize changes"

Kay keeps native Code agents, skills, subagents, and AGENTS.md support in play. Sidekick adds the packaging, compatibility aliases, session-scoped marker, hook summaries, and project-local .kay/conversations.idx lookup surface.

AGENTS.md Bootstrap

On your first /forge invocation, if no non-empty AGENTS.md exists in the project root, Sidekick creates one. This file accumulates standing instructions for Forge — project conventions, preferences, and patterns the host AI discovers over time.

The bootstrap file contains the default Forge output format expectations (STATUS, FILES_CHANGED, ASSUMPTIONS, PATTERNS_DISCOVERED) and delegation principles. You can edit it at any time to add project-specific rules.

AGENTS.md tiers

Sidekick maintains instructions at three levels: Global (~/forge/AGENTS.md — cross-project), Project (./AGENTS.md — this repo), and Session log (docs/sessions/ — per-session evolution). The host writes only new, deduplicated instructions; L3 takeover updates are proposed for user confirmation before AGENTS.md is changed.
