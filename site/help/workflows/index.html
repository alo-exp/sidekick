Workflows

Use one workflow shape for both supported sidekicks: activate, delegate, observe, verify, retry if needed, then stop or switch.

Kay Workflow

1
Activate Kay.

Run /sidekick:kay-delegate, optionally with xiaomi or ocg. The host records Kay as the active sidekick.

2
Run the child runtime.

Active Kay mode delegates implementation through kay exec. The hook layer keeps the run bounded and surfaces compact task progress.

3
Review and verify.

The host reviews changed files, checks assumptions, runs tests, and compares the result to the original request.

/sidekick:kay-delegate kay exec "<bounded task>" /sidekick:kay-stop

Codex Workflow

1
Activate Codex.

Run /sidekick:codex-delegate. Sidekick verifies the local OpenAI Codex CLI path before handing over work.

2
Run the child runtime.

Active Codex mode delegates through codex exec using gpt-5.4-mini with extra-high reasoning.

3
Review and verify.

The host checks the diff, tests, integration behavior, and any assumptions before reporting completion.

/sidekick:codex-delegate codex exec --model gpt-5.4-mini --reasoning-effort xhigh "<bounded task>" /sidekick:codex-stop

Retry Workflow

If verification finds a missed requirement, regression, wrong file, syntax error, API failure, or incomplete trial, the host should relaunch the active sidekick with a narrower prompt that names the failed check and the expected final state.

Rule: Do not accept sidekick output because it says success. Accept it only after host verification confirms the requested behavior in the repository.

Switching Sidekicks

Stop the active sidekick first, then activate the other one. This keeps the active-sidekick selector unambiguous and prevents mixed runtime state in the same host session.

/sidekick:kay-stop /sidekick:codex-delegate
