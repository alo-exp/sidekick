Concepts

Sidekick is a delegation layer. The host AI keeps the plan, conversation, review, and final correctness. The active sidekick performs bounded implementation work through its native runtime.

Host And Sidekick

Host AI

Plans the task, communicates with the user, selects the active sidekick, reviews output, runs verification, and decides whether more work is needed.

Sidekick

Runs the delegated task as a child runtime. Kay uses kay exec. Codex uses codex exec.

Supported Sidekicks

Kay: installed and repaired through Sidekick, with OpenCode Go as the default route. Session selectors can choose xiaomi or force ocg.

Codex: delegates to the local OpenAI Codex CLI, using gpt-5.4-mini with extra-high reasoning.

active-sidekick

The shared selector lives under the host session root and contains either kay or codex. It prevents both sidekicks from being active in the same host session and lets hooks know which runtime boundary to apply.

~/.sidekick/sessions/<session>/active-sidekick .kay/sessions/<session>/.kay-delegation-active .codex/sessions/<session>/.codex-delegation-active

Hook Boundary

When a sidekick is active, Sidekick hooks guard direct host mutations and route supported child-runtime commands through bounded progress surfaces. The hook is a safety boundary, not a replacement for review.

Host Verification

Every delegated task needs a host verification pass. The host checks for missed requirements, integration errors, regressions, wrong logic, syntax errors, wrong files, unverified assumptions, knowledge gaps, misunderstood task scope, incomplete trials, provider API failures, and external execution failures.

Generated Host Bundles

Canonical skills live in skills/. Generated host bundles under agents/claude/ and agents/codex/ are rendered by the sync script and should stay aligned with the canonical source.
