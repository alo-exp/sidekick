# Knowledge — 2026-06

## Kay runtime context (Sidekick live testing)

- **Kay** (`alo-labs/kay`) is a fork of industry-standard **Codex OSS**, originally derived from **every/code**, with a small set of additional features plus extra model providers/models (e.g. OpenCode Go / `mimo-v2.5-pro`).
- When filing Kay issues from Sidekick delegation tests, triage each finding into:
  1. **Third-party model/provider compatibility** — behavior OSS Codex was not designed for (e.g. `mimo-v2.5-pro` tool-call quirks).
  2. **Upstream Codex OSS** — likely shared with stock Codex.
  3. **every/code / Kay fork** — regression or behavior specific to the Kay lineage.
- Sidekick live harness: `tests/run_kay_live_task.bash`, prompts under `tests/kay-live-prompts/`, secrets in gitignored `.env.local`.
- **SB stop hook (Kay harness sessions):** Stop-tier enforcement reads `~/.cursor/.silver-bullet/state` (Cursor) and `.silver-bullet.json` `skills.required_planning`. Legacy config (`config_version` < 0.39.3) merges repo overrides with the modern planning floor (`silver-quality-gates`, `silver-context`, `silver-plan`) and can block harness-only work. Harness/QA sessions should keep `required_planning: ["silver-quality-gates"]` only; feature/release work re-adds `silver-context` + `silver-plan`. Completed composed workflows must be under `.planning/workflows/.archive/` (not `*.md` in the workflows root) or `workflow-chain-guard` blocks edits when multiple tickets exist. Immediate unblock without config change: `printf 'silver-quality-gates\nsilver-context\nsilver-plan\n' > ~/.cursor/.silver-bullet/state` or invoke `/silver:fast` at session start. `.planning/config.json` `mode: yolo` does **not** bypass the stop hook.
