Supported Sidekicks

Kay and Codex only.

The public surface now matches the plugin registry: Sidekick supports Kay and Codex. Both run behind the same host-owned verification contract and the shared active-sidekick selector.

Kay sidekick

Use Kay when you want Sidekick to route implementation through the Kay runtime and its provider selection layer.

  • kay exec is the delegate command.
  • /sidekick:kay-delegate xiaomi selects Xiaomi routing.
  • /sidekick:kay-delegate ocg forces OpenCode Go routing.
  • code and coder are compatibility aliases.

Codex sidekick

Use Codex when you want a local OpenAI Codex CLI child process to execute the delegated task.

  • codex exec is the delegate command.
  • Model routing is fixed to gpt-5.4-mini.
  • Reasoning effort is extra-high.
  • Kay aliases are rejected in Codex mode.

Host verification

The host never accepts a sidekick result only because the sidekick reports success.

  • Diffs, tests, requirements, and integration points are checked.
  • Missed requirements trigger a focused relaunch.
  • Only one sidekick can be active in a host session.
  • Stop commands clear active delegation markers.
Runtime Contract

The host stays accountable.

Sidekick is not an autopilot. It is a delegation harness: the host AI owns planning, user communication, review, and final correctness.

01

Plan

The host scopes the task and creates a bounded prompt for the active sidekick.

02

Delegate

Hooks route mutating work through Kay or Codex while that sidekick is active.

03

Verify

The host checks the result against requirements, tests, integration behavior, and file scope.

04

Recover

If verification fails, the host relaunches the active sidekick with focused guidance.
