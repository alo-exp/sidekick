The Problem

Delegation Without Verification Is Theater

You did not adopt a host AI to merge unreviewed sidekick output. You adopted it to move faster with accountability. Most teams hit the same walls when implementation leaves the host's hands.

PAIN 01

The Host Edits Files While a Sidekick Is Active

PreToolUse enforcement is absent — the host bypasses the active sidekick mid-task and two writers collide in the same repo.

PAIN 02

“Done” Means the Child Runtime Said So

The sidekick reports success. Tests were not re-run. Integration points were not checked. The PR is open anyway.

PAIN 03

Kay and Codex State Collides

Without a session-scoped active-sidekick marker, hosts lose track of which runtime owns implementation.

PAIN 04

Progress Output Floods Context

Raw child logs land in the host session. Bounded, redacted summaries never arrive — so the host cannot steer.

PAIN 05

Every Host Reinvents Delegation

Claude Code, Codex, and Cursor each need the same brain/hands split — but copy-pasted prompts do not enforce it.

PAIN 06

Frontier Models Burn on Implementation

Expensive host models write boilerplate that Kay or Codex could run — if delegation were reliable and verified.

Financial

Frontier token spend on implementation plus rework when unverified diffs land in main.

Time

Hosts re-run tests and re-review PRs the sidekick already declared finished.

Opportunity

Teams that verify delegation ship trust; teams on vibes accumulate silent regressions.

Emotional

Review fatigue when “agent done” cannot be believed without opening every file.

It does not have to be heavyweight for every typo — but delegated implementation needs a system that pushes back.

What If

The Host Stays the Brain. The Sidekick Stays the Hands.

Same models and hosts you already use — with a enforced split between planning and implementation.

You open Monday's queue: every sidekick task has a verification record. The host compared diffs to the original prompt, ran tests, and relaunched Kay or Codex when a requirement was missed. Implementation ran on a cheaper runtime; the host never merged on faith.

You describe the task. The host shapes the prompt and activates /sidekick:kay or /sidekick:codex. Hooks block host edits while the sidekick works. Bounded progress surfaces through PostToolUse. The host verifies — then reports done to the user. If verification fails, the host relaunches with correction context until the failure is resolved.

How It Works

Host-Owned Verification, Not Hope

We built Sidekick because child runtimes are fast — but a sidekick task is not done until the host verifies it against the original prompt and repository behavior.

FAQ

Common Objections

“I already use Task / subagents.”

Subagents do not session-gate host edits or standardize Kay/Codex progress across Claude, Codex, and Cursor. Sidekick is the enforcement layer.

“Another plugin to install.”

One marketplace install per host. Core delegation uses your existing Kay and Codex CLIs — Sidekick does not replace them.

“Which sidekick should I use?”

Kay for OpenCode Go / Xiaomi routing; Codex when you want the local Codex CLI. Only one is active at a time — by design.
