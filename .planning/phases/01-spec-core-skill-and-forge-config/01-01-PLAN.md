**## 1. Overview and Mental Model**
- Claude = Brain (plan, communicate, review), Forge = Hands (write, edit, run, commit, test)
- When `/forge` is active, Claude is a thin orchestrator; Forge does 100% of implementation
- Reference that `skills/forge.md` already embodies this model (line 27-29)

**## 2. Activation Protocol**
- User invokes `/forge` which triggers `skills/forge/SKILL.md`
- Health check (all 4 must pass per D-03):
  1. `~/.local/bin/forge` exists OR `forge` is on PATH
  2. `forge info` exits 0 and output contains a provider name
  3. `~/forge/.credentials.json` exists with non-empty `api_key`
  4. `~/forge/.forge.toml` contains non-empty `provider_id` and `model_id`
- If ANY fail: print error referencing `skills/forge.md` STEP 0A (do NOT duplicate install instructions)
- If all pass: create zero-byte marker `~/.claude/.forge-delegation-active`
- Bootstrap `.forge/` config files if absent (details in section 11)
- Confirm to user: "Forge-first delegation mode activated. All implementation tasks will be routed to Forge."
- If marker already exists: re-run health check, acknowledge "already active" if healthy

**## 3. Deactivation Protocol**
- User invokes `/forge:deactivate`
- Delete `~/.claude/.forge-delegation-active`
- Confirm: "Forge-first mode deactivated. Claude-direct mode restored."
- If marker doesn't exist: acknowledge "Forge-first mode is not currently active."

**## 4. Task Prompt Format**
- Per D-09, every Claude-to-Forge prompt uses this exact structure:
```
OBJECTIVE: [one sentence — what must be true when done]
CONTEXT: [files involved, current state of relevant code]
DESIRED STATE: [concrete output — what the file/function/result should look like]
SUCCESS CRITERIA: [testable conditions — how Forge knows it succeeded]
INJECTED SKILLS: [list of skills injected to .forge/skills/ for this task]
```
- Maximum prompt length: 2,000 tokens
- Omit conversation history entirely
- Reference that `skills/forge.md` STEP 3 already has prompt crafting guidance; this format is the formalized contract

**## 5. Delegation Loop**
- Step 1: Claude checks marker file exists (delegation mode active)
- Step 2: Claude composes task prompt per section 4 format
- Step 3: Claude injects relevant skills to `.forge/skills/` (see section 8)
- Step 4: Claude submits task to Forge (reference `skills/forge.md` STEP 4)
- Step 5: Claude monitors Forge output for completion or failure signals (reference STEP 6)
- Step 6: Claude communicates outcome to user in plain language
- DLGT-04 enforcement: While marker exists, Claude MUST NOT use Write, Edit, or Bash tools for implementation (except Level 3 fallback)

**## 6. Failure Detection Criteria**
- Error signals: Forge prints an explicit error message or exits non-zero
- Wrong output: Forge produces output that doesn't match SUCCESS CRITERIA (repeated 2+ times)
- Stall condition: Forge produces no meaningful output for an extended period
- Reference `skills/forge.md` STEP 5 for existing failure recovery patterns
- These criteria feed into the Fallback Ladder (section 7)

**## 7. Fallback Ladder Contract**
- Level 1 — Guide: On failure detection, Claude reframes the prompt with clarifying context and retries. Add specifics about what Forge likely misunderstood. Single retry.
- Level 2 — Handhold: If Level 1 fails, Claude decomposes the task into subtasks. Submit each subtask sequentially with tighter scoping. Max 3 subtask attempts.
- Level 3 — Take over: If Level 2 fails, Claude performs the task directly using Write/Edit/Bash. After completion, produce a debrief:
  - What the task was
  - Why Forge failed (diagnosis)
  - What was learned
  - Proposed AGENTS.md update to prevent recurrence
- Note: Full implementation in Phase 2. This section defines the contract.

**## 8. Skill Injection Protocol**
- Mapping table (Claude skill -> Forge SKILL.md equivalent):
  - Silver Bullet quality-gates -> `.forge/skills/quality-gates/SKILL.md`
  - Silver Bullet security -> `.forge/skills/security/SKILL.md`
  - Engineering testing-strategy -> `.forge/skills/testing-strategy/SKILL.md`
  - Engineering code-review -> `.forge/skills/code-review/SKILL.md`
- Selective injection: Only inject skills matching the task type (e.g., testing task gets testing-strategy)
- Forge SKILL.md format per D-10:
  - YAML frontmatter: `id`, `title`, `description`, `trigger`
  - No `Skill tool` references
  - No `AskUserQuestion` calls
  - No Claude-specific tool names (Read/Edit/Write -> generic file operation language)
  - Imperative language: "Run X", "Write Y", "Verify Z"
  - `trigger` field: keywords that cause Forge Skill Engine auto-detection
- Note: Full injection automation in Phase 2 (SINJ-01-05). This section defines format and mapping.

**## 9. AGENTS.md Write Protocol**
- What to extract: corrections, user preferences, project patterns, Forge behavior observations
- Global tier: Append to `~/forge/AGENTS.md` (cross-project, cross-session)
- Project tier: Append to `./AGENTS.md` (project-specific)
- Session log: Write to `docs/sessions/` capturing instruction evolution
- Deduplication algorithm: Before every write, scan existing AGENTS.md content. If semantically equivalent instruction exists, skip. Use exact substring match as primary check, semantic similarity as secondary.
- Format: Follow Forge's recommended format — action-oriented, specific, organized by category
- Bootstrap: On first invocation with empty AGENTS.md, populate from `skills/forge.md` content
- Note: Full implementation in Phase 3 (AGNT-01-08). This section defines the protocol.

**## 10. Token Budget Rules**
- Max task prompt: 2,000 tokens (per D-09)
- Omit: Full conversation history, unrelated file contents, verbose error logs
- Include: Only files directly relevant to the task, current state of target files
- Compaction triggers: `.forge.toml` settings (see section 11)
- Skill injection: Only relevant skills, not all available skills
- Note: Full optimization in Phase 3 (TOKN-01-04). This section defines the constraints.

**## 11. Forge Config File Specs**
- **`.forge/agents/forge.md`** (per D-06):
  - Content: AGENTS.md reference, delegation-mode awareness, structured output format expectations (STATUS/FILES_CHANGED/ASSUMPTIONS/PATTERNS_DISCOVERED), standing instruction to update AGENTS.md
  - Created on first `/forge` invocation if absent; NEVER overwritten (D-05)
- **`.forge.toml`** (per D-07):
  - `max_tokens = 16384`
  - `[compact]` section: `token_threshold = 80000`, `eviction_window = 0.20`, `retention_window = 6`
  - `[session]` section: `provider_id` and `model_id` (user-configurable defaults)
  - Created on first `/forge` invocation if absent; NEVER overwritten (D-05)
- **`.forge/skills/`** (per D-08):
  - Bootstrap set: quality-gates, security, testing-strategy, code-review
  - Each at `.forge/skills/<name>/SKILL.md`
  - Created on first `/forge` invocation if directory absent
  - Individual skill files: created if absent, never overwritten

**## 12. Composition Contract**
- `skills/forge.md` (existing, 862 lines): Always-on orchestration protocol. Handles delegation decisions (STEP 1), project context (STEP 2), prompt crafting (STEP 3), running Forge (STEP 4), failure recovery (STEP 5), post-delegation review (STEP 6).
- `skills/forge/SKILL.md` (new, ~50-100 lines): User-invoked mode switch. Handles activation, health check, config bootstrap, deactivation, session state.
- Relationship: SKILL.md activates the mode; forge.md executes within it. SKILL.md does NOT duplicate any forge.md logic.
- Flow: `/forge` -> SKILL.md activates -> marker set -> subsequent tasks -> forge.md orchestrates -> `/forge:deactivate` -> SKILL.md deactivates -> marker deleted
